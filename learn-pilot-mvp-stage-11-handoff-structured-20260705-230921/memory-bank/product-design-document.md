# AI 学习计划软件 App PRD 产品需求文档

版本：V1.0  
适用端：移动端 App  
产品阶段：个人自用 MVP  
文档目标：用于原型设计、开发评审、作品集整理与 AI 产品经理项目展示

## 假设条件

- 本产品第一阶段仅服务个人学习，不考虑商业化、付费订阅、拉新增长、社区运营、多人协作。
- MVP 默认使用移动端优先设计，核心使用场景为碎片时间查看任务、晚上集中学习、周末复盘。
- AI 能力优先用于规划、拆解、总结、调整，不承诺绝对学习效果。
- 资源推荐优先输出资源类型、筛选规则和可核验来源，不编造具体课程、书籍或网站。
- 第一版可先接入通用大模型 API，资料搜索和真实资源校验可作为后续增强。

# 1. 产品概述

## 1.1 产品名称建议

| 名称 | 命名理由 |
| --- | --- |
| 学径 AI | 强调“学习路径”，适合定位为个人学习路线规划工具。 |
| 计划知行 | 强调从计划到执行，覆盖路线生成、每日任务和复盘闭环。 |
| LearnPilot | “学习驾驶舱/领航员”含义，突出 AI 辅助规划与动态调整。 |

## 1.2 一句话产品定位

面向个人学习者的 AI 学习计划 App，帮助用户把一个新领域拆成可执行、可追踪、可复盘的移动端学习计划。

## 1.3 产品背景

个人学习者在面对 AI、数据分析、产品经理、考公、科研写作、英语等新方向时，常见问题不是“没有资料”，而是资料过多、路径不清、任务不可执行、进度难以持续、学习效果难判断。该 App 通过 AI 收集目标、基础、时间和偏好，生成个性化学习路线图，并进一步拆解为每日任务、资源建议、阶段产出和复盘调整。

## 1.4 个人学习痛点

| 痛点 | 具体表现 | 产品解决方式 |
| --- | --- | --- |
| 路径不清 | 不知道先学什么、后学什么 | AI 生成阶段化学习路线图 |
| 资料过载 | 收藏很多资料但不知道选哪个 | 每阶段限制推荐数量并标注适配理由 |
| 计划空泛 | 计划停留在“学习 XX” | 拆成每日任务、预计耗时、完成标准 |
| 执行困难 | 连续几天未完成后计划崩盘 | 支持延期、跳过、重排和自动降载 |
| 缺少反馈 | 不知道是否真正掌握 | 阶段测试、产出物、AI 复盘和掌握度判断 |
| 复盘缺失 | 学完没有总结沉淀 | 笔记整理、周复盘、下一步建议 |

## 1.5 产品目标

- 让用户在 10 分钟内完成一个可执行学习计划的创建。
- 将长期学习目标拆解为阶段目标和每日任务。
- 帮助用户识别合适资料，减少无效收藏。
- 追踪任务、时长、进度、产出和复盘。
- 根据执行情况动态调整学习计划。
- 用阶段测试和产出物判断用户是否真正掌握阶段目标。

## 1.6 核心使用场景

- 用户想在 30 天内入门 AI 产品经理。
- 用户准备转行数据分析，需要 8 周学习路线。
- 用户准备考公，需要按考试模块安排每日复习。
- 用户正在入门科研写作，需要论文阅读、写作练习和阶段产出。
- 用户学习英语，希望按听说读写和考试目标拆分计划。

## 1.7 产品差异化价值

- 从“学习资料推荐”升级为“目标-路线-任务-执行-复盘”的闭环。
- 每日任务具备明确完成标准，避免计划不可执行。
- AI 会根据用户真实执行情况调整计划，不是一次性生成静态清单。
- 强调移动端轻量使用，适合每天查看、打卡、记录和复盘。

## 1.8 当前版本边界

当前版本明确不做：商业化、付费、订阅、拉新、社群运营、好友排名、内容广场、多人协作、复杂后台管理。

# 2. 目标用户分析

## 2.1 核心用户类型

- 想快速入门 AI、数据分析、产品经理等方向的大学生。
- 想转行或找实习的学生。
- 想系统学习研究方向的研究生。
- 想提高某项技能的职场新人。
- 想备考或自学但缺少计划的人。

## 2.2 典型用户画像

| 用户类型 | 学习目标 | 当前痛点 | 使用动机 | 关键需求 | 对 App 的期待 |
| --- | --- | --- | --- | --- | --- |
| 大学生 | 30 天入门 AI 产品经理 | 不知道岗位能力模型和学习顺序 | 做求职项目和作品集 | 路线图、资料、项目产出 | 能告诉我每天学什么、最后产出什么 |
| 转行学生 | 8 周掌握数据分析基础 | 资料多但缺少项目练习 | 找实习、补能力 | 阶段任务、练习项目、进度追踪 | 能把技能转成可展示作品 |
| 研究生 | 系统进入新研究方向 | 文献难筛、方向抽象 | 建立研究框架 | 文献阅读计划、笔记、复盘 | 能沉淀研究问题和知识结构 |
| 职场新人 | 提升产品/运营/英语能力 | 工作后时间碎片化 | 提高职业竞争力 | 每日短任务、提醒、周复盘 | 不用花很多时间做计划 |
| 备考用户 | 准备考试或证书 | 进度容易失控 | 稳定执行复习计划 | 模块拆分、错题/测试、重排计划 | 能根据落后情况自动调整 |

## 2.3 使用前后差异

| 阶段 | 使用前 | 使用后 |
| --- | --- | --- |
| 明确目标 | 目标模糊，如“想学 AI” | 目标被拆成周期、阶段、产出和评估标准 |
| 选择资料 | 靠搜索和收藏 | 按阶段、难度、时长和适配理由推荐 |
| 每日执行 | 不知道今天学什么 | 首页直接展示今日任务和完成标准 |
| 学习反馈 | 只知道“学过” | 通过任务完成率、产出物、测试和复盘判断掌握情况 |
| 计划调整 | 一旦落后就放弃 | 自动降载、重排、延期或调整目标范围 |

## 2.4 用户最在意的功能点

- 计划是否适合自己的基础。
- 今日任务是否具体可执行。
- 资料是否可靠且不过量。
- 落后时是否能自动调整。
- 能否判断自己是否真正掌握。

# 5. 核心功能详细说明

## 5.1 功能一：学习目标创建

| 项目 | 说明 |
| --- | --- |
| 功能目标 | 让用户用最少输入描述学习目标，并生成后续测评和路线图所需的结构化信息。 |
| 用户入口 | 首页“新建学习计划”按钮、空状态页“开始制定计划”。 |
| 用户操作流程 | 点击新建计划 -> 输入学习方向 -> 选择目标类型 -> 选择周期和每日时间 -> 填写学习偏好 -> 点击“生成学习方案”。 |
| 输入字段 | 学习方向、具体目标、当前基础自评、学习周期、每日可用时间、每周可学习天数、偏好资料类型、目标产出、截止日期、备注。 |
| 可选周期 | 7 天、14 天、30 天、60 天、90 天、自定义。 |
| 每日学习时间 | 15 分钟、30 分钟、60 分钟、90 分钟、120 分钟、自定义。 |
| 目标类型 | 入门了解、系统学习、考试备考、求职项目、科研方向、语言提升、技能提升。 |
| 系统处理逻辑 | 校验目标是否过宽；将自然语言目标结构化；判断周期和目标是否匹配；必要时提示缩小范围。 |
| 输出结果 | 学习目标卡片、测评建议、预估计划难度、是否需要调整目标范围。 |
| 异常情况 | 目标过大、周期过短、每日时间过少、输入为空、网络或 AI 生成失败。 |
| 验收标准 | 用户可在 3 分钟内完成表单；必填字段缺失时有提示；生成前展示确认信息；目标过大时给出缩小建议。 |
| 优先级 | P0 |

示例输入：

| 字段 | 示例 |
| --- | --- |
| 学习方向 | AI 产品经理 |
| 具体目标 | 30 天内入门，并能做一个 AI App PRD 作品集项目 |
| 当前基础 | 了解产品经理基础，不熟悉 AI 技术 |
| 学习周期 | 30 天 |
| 每日时间 | 60 分钟 |
| 偏好方式 | 文章、案例、项目实践 |

示例输出：

| 输出项 | 内容 |
| --- | --- |
| 目标判断 | 目标可行，但需聚焦“AI 产品基础 + PRD 项目实践” |
| 建议路径 | AI 基础认知 -> AI 产品方法 -> 案例分析 -> PRD 输出 -> 复盘优化 |
| 下一步 | 建议完成 8 分钟基础测评 |

## 5.2 功能二：用户基础测评

| 项目 | 说明 |
| --- | --- |
| 功能目标 | 判断用户当前基础，避免计划过难或过浅。 |
| 用户入口 | 创建学习计划后自动进入；路线图页可重新测评。 |
| 测评方式 | 自评 + 选择题 + 情景题 + 简短主观题。MVP 可先使用自评和 5-8 道题。 |
| 测评题类型 | 概念理解题、应用判断题、任务拆解题、案例分析题、学习经验自评。 |
| 用户自评 | 零基础、初级、中级、进阶，并可填写已学内容。 |
| 跳过处理 | 用户可跳过，系统默认按“初级偏保守”生成计划，并标记可信度较低。 |
| AI 调整逻辑 | 根据得分、主观描述和目标难度，调整阶段起点、任务难度、资料难度和每日任务量。 |
| 输出结果 | 基础等级、优势点、薄弱点、计划调整建议。 |
| 验收标准 | 完成测评后能输出等级；跳过时有默认策略；测评结果会影响路线图难度。 |
| 优先级 | P0 |

水平定义：

| 水平 | 定义 | 计划策略 |
| --- | --- | --- |
| 零基础 | 几乎不了解该领域术语和基本概念 | 增加概念入门和低门槛资料 |
| 初级 | 知道部分概念，但缺少系统框架和实践 | 重点补框架和基础任务 |
| 中级 | 有相关学习或项目经验 | 减少基础内容，增加案例和产出 |
| 进阶 | 能独立完成任务，希望提升深度 | 增加复杂项目、论文/文档和评估任务 |

示例测评题：

| 题目 | 类型 | 目的 |
| --- | --- | --- |
| 你如何理解大语言模型在产品中的作用？ | 主观题 | 判断 AI 产品认知 |
| 以下哪类任务更适合用 RAG 方案？ | 选择题 | 判断基础技术理解 |
| 如果要设计一个 AI 学习助手，你会先定义哪些用户需求？ | 情景题 | 判断产品思维 |

## 5.3 功能三：AI 学习路线图生成

| 项目 | 说明 |
| --- | --- |
| 功能目标 | 将用户目标拆成阶段、内容、资料、任务、产出和验收方式。 |
| 用户入口 | 完成目标创建和测评后自动生成；计划详情页点击“重新生成”。 |
| 生成逻辑 | 解析目标 -> 判断基础 -> 估算可用总时长 -> 划分阶段 -> 分配学习内容 -> 设置阶段产出 -> 拆每日任务 -> 生成验收方式。 |
| 阶段划分 | 通常 3-5 个阶段，每阶段包含目标、内容、资料类型、任务节奏、产出物、测试方式。 |
| 防空泛机制 | 每阶段必须包含可交付产出；每日任务必须有动作动词、预计耗时、完成标准。 |
| 适配基础机制 | 使用测评等级决定起点、难度、资料类型和任务密度。 |
| 手动调整 | 用户可编辑阶段名称、时间范围、任务数量、资料偏好、目标产出。 |
| 异常处理 | 目标过大时提示缩小；周期过短时给压缩版或建议延长；基础过弱时加入预备阶段。 |
| 验收标准 | 路线图至少包含 3 个阶段；每阶段有目标、内容、产出、验收；任务能落到每日。 |
| 优先级 | P0 |

| 阶段名称 | 时间范围 | 阶段目标 | 核心学习内容 | 推荐资料类型 | 每日任务示例 | 阶段产出 | 阶段验收方式 |
| --- | --- | --- | --- | --- | --- | --- | --- |

## 5.4 功能四：学习资料推荐

| 项目 | 说明 |
| --- | --- |
| 功能目标 | 为每个阶段推荐少量、高适配、可执行的学习资料。 |
| 资源类型 | 视频课、书籍、文章、论文、项目案例、官方文档、开源项目、练习题、工具教程。 |
| 推荐依据 | 用户目标、当前基础、学习阶段、难度、权威性、时长、可获得性、资料类型偏好。 |
| 展示信息 | 标题、资源类型、适合阶段、预计耗时、难度、推荐理由、来源链接、是否需核验、用户标记。 |
| 筛选规则 | 按类型、难度、耗时、阶段、已学/想学/不适合筛选。 |
| 防过载机制 | 每阶段默认推荐 3-5 条主资源和 2 条补充资源，不做无限列表。 |
| 用户标记 | 已学、想学、不适合、失效、太难、太简单。 |
| 异常处理 | 链接失效则提示更换；质量低则降低权重；难度不匹配则重新推荐。 |
| 验收标准 | 每条资源有推荐理由；不编造不存在资源；不确定资源标注“需要进一步核验”。 |
| 优先级 | P0 |

资源真实性规则：

- 示例资源可以作为“资源类型示例”展示。
- 涉及具体书籍、课程、网站、论文时，必须标注来源或“需要进一步核验”。
- AI 不得生成无法验证的链接。
- 对考试、政策、学术类资料需提示时效性。

## 5.5 功能五：每日学习任务

| 项目 | 说明 |
| --- | --- |
| 功能目标 | 将路线图拆为用户当天可直接执行的任务。 |
| 用户入口 | 首页今日任务卡片、底部 Tab“任务”。 |
| 生成逻辑 | 根据当前阶段、剩余天数、每日可用时间、历史完成情况生成 2-4 个任务。 |
| 任务类型 | 阅读、观看、练习、输出、测试、复盘、资料整理、项目实践。 |
| 预计耗时 | 每个任务 10-45 分钟；每日总时长不超过用户设定时间的 120%。 |
| 完成标准 | 必须明确，如“写 150 字总结”“完成 5 道题”“提交一份案例表”。 |
| 难度 | 简单、适中、挑战；根据基础和连续完成情况调整。 |
| 学习产出 | 笔记、截图、链接、文本总结、测验结果、项目文档。 |
| 未完成处理 | 可延期、跳过、拆小、重排；连续 3 天未完成触发计划降载建议。 |
| 提前完成处理 | 提供加餐任务、提前解锁下一任务或保留休息。 |
| 验收标准 | 今日任务可一屏查看；任务状态可更新；延期/跳过会影响后续计划。 |
| 优先级 | P0 |

## 5.6 功能六：学习进度追踪和复盘

| 项目 | 说明 |
| --- | --- |
| 功能目标 | 让用户清楚看到执行状态、阶段完成情况和下一步调整建议。 |
| 展示方式 | 首页进度环、任务完成率、连续打卡天数、阶段进度条、学习时长趋势。 |
| 打卡机制 | 完成至少 1 个核心任务后可打卡；仅打开 App 不算打卡。 |
| 任务完成率 | 已完成任务数 / 应完成任务数。 |
| 学习时长 | 用户手动记录或任务完成时自动累加预计时长，可手动修正。 |
| 周复盘 | 每 7 天或阶段结束触发，收集完成率、难点、满意度、下周时间变化。 |
| AI 总结 | 总结已完成内容、薄弱点、执行风险、下一周建议。 |
| 掌握判断 | 综合任务产出、阶段测试、用户自评、AI 对笔记/答案的评估。 |
| 调整建议 | 降低任务量、补基础、延长周期、替换资料、调整目标范围。 |
| 验收标准 | 用户能查看进度；能完成周复盘；AI 建议需用户确认后生效。 |
| 优先级 | P1 |

移动端设计原则：

- 首页优先显示“今天要做什么”，避免信息过载。
- 底部 Tab 建议为：首页、任务、路线、笔记、我的。
- 关键按钮固定在页面底部，减少单手操作成本。
- 复杂信息使用分层折叠，如阶段详情、资源说明、AI 调整原因。

# 7. 核心用户流程

主流程：

用户进入 App  
→ 查看首页当前学习状态  
→ 点击“新建学习计划”  
→ 输入学习方向、目标、周期、每日可用时间、学习偏好  
→ 系统提示目标是否合理  
→ 用户确认并进入基础测评  
→ 完成自评和题目  
→ AI 生成基础等级和薄弱点  
→ AI 生成学习路线图  
→ 用户查看阶段目标、资料类型、阶段产出  
→ 用户确认或手动调整路线图  
→ 系统生成今日任务  
→ 用户进入每日任务页  
→ 按任务要求学习并提交笔记或产出  
→ 系统记录完成状态、时长和进度  
→ AI 总结当天学习状态  
→ 用户每周进入复盘页  
→ 系统根据完成率、笔记、测试结果和用户反馈给出调整建议  
→ 用户确认调整  
→ 系统重排后续计划。

异常流程：

| 异常情况 | 处理流程 |
| --- | --- |
| 用户跳过测评 | 系统提示“计划准确度会降低” -> 默认按初级偏保守生成 -> 后续根据任务表现修正基础等级。 |
| 用户目标过大 | 系统识别目标范围过宽 -> 给出 2-3 个缩小方向 -> 用户选择“压缩目标”或“延长周期”。 |
| 连续多天未完成任务 | 第 2 天提醒补做或延期 -> 第 3 天触发降载建议 -> 用户确认后减少每日任务量或延长周期。 |
| 推荐资料不合适 | 用户点击“不适合” -> 选择原因：太难、太简单、质量低、链接失效 -> 系统重新推荐并记录偏好。 |
| 临时修改学习周期 | 用户进入计划设置 -> 修改截止日期或每日时间 -> AI 重新计算剩余任务 -> 用户确认后生效。 |

# 8. AI 功能设计

| AI 功能名称 | 用户输入 | 系统补充数据 | AI 处理逻辑 | 输出结果 | 用户可编辑项 | 风险点 | 风险控制 | 验收标准 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| AI 生成学习路线图 | 目标、基础、周期、时间 | 测评结果、偏好、历史完成率 | 阶段划分、任务密度估算、产出设计 | 阶段化路线图 | 阶段、时间、产出、任务量 | 计划不准确、过空泛 | 强制输出阶段目标、任务、产出、验收 | 路线图可拆成每日任务 |
| AI 推荐资料 | 学习阶段、偏好 | 难度、耗时、资源标记 | 按适配度推荐少量资源 | 资源列表和推荐理由 | 标记状态、替换资源 | 编造资源、链接失效 | 标注来源/类型/需核验，不生成假链接 | 每条有理由和核验状态 |
| AI 拆解每日任务 | 路线图、当天时间 | 剩余任务、历史完成情况 | 控制任务数量和耗时 | 今日任务列表 | 延期、跳过、拆小 | 任务过多或不可执行 | 每任务必须有完成标准 | 总耗时不超设定 |
| AI 调整计划 | 完成率、反馈 | 连续未完成天数、阶段进度 | 判断降载、延期、补基础 | 调整建议 | 是否应用、调整幅度 | 频繁调整导致混乱 | 用户确认后生效，保留版本 | 调整前后可对比 |
| AI 总结学习笔记 | 笔记文本 | 关联任务、阶段目标 | 提炼知识点、问题、行动项 | 摘要和知识标签 | 摘要、标签 | 总结失真 | 保留原文，允许编辑 | 摘要不替代原笔记 |
| AI 生成阶段测试题 | 阶段内容 | 任务和资料记录 | 生成题目和评分标准 | 测试题、参考答案 | 题目数量、难度 | 题目质量不稳定 | 标注参考性质，用户可跳过 | 能覆盖阶段目标 |
| AI 判断掌握度 | 答案、笔记、产出 | 完成率、测试得分 | 综合评分和薄弱点识别 | 掌握度等级 | 自评修正 | 误判掌握情况 | 不给绝对结论，用建议语气 | 输出依据清楚 |
| AI 下一步建议 | 复盘结果 | 计划剩余内容 | 生成后续学习策略 | 补学、加速、调整建议 | 接受/拒绝 | 建议过泛 | 建议必须绑定具体任务 | 可一键应用到计划 |

AI 生成内容风险控制：

- 不确定内容必须提示用户核验。
- 推荐资料必须标注来源、类型或“需要进一步核验”。
- 学习路线图允许用户修改，不作为唯一正确方案。
- AI 不承诺绝对学习效果，只提供辅助建议。
- 不编造不存在的课程、论文、链接和案例。
- 考试、政策、学术内容需提示资料时效性。
- 所有 AI 调整建议默认“待确认”，不得自动覆盖用户计划。

# 10. MVP 版本规划

## 10.1 MVP 核心目标

MVP 验证的核心假设：当 AI 能把个人学习目标转化为阶段路线、每日任务、资源建议和复盘调整时，用户是否更容易持续执行，并更清楚自己是否掌握阶段目标。

## 10.2 MVP 功能范围

| 功能模块 | 是否进入 MVP | 原因 | 优先级 |
| --- | --- | --- | --- |
| 学习目标创建 | 是 | 全流程起点 | P0 |
| 基础测评 | 是 | 决定计划个性化质量 | P0 |
| AI 路线图生成 | 是 | 核心价值 | P0 |
| 每日任务 | 是 | 决定执行效果 | P0 |
| 资料推荐 | 是 | 解决资料选择问题 | P0 |
| 进度追踪 | 是 | 提供反馈 | P0 |
| 笔记记录 | 是，简化版 | 支持学习产出 | P1 |
| 周复盘和调整 | 是，简化版 | 形成闭环 | P1 |
| 提醒和打卡 | 是，基础版 | 提升执行 | P1 |
| 个人学习档案 | 否，保留简单历史 | 非首要闭环 | P2 |

## 10.3 暂不做功能

- 商业化功能。
- 付费订阅。
- 社区功能。
- 好友排名。
- 复杂运营活动。
- 公开内容广场。
- 多人协作学习。
- 复杂后台管理。
- 自动抓取全网资料库。
- 多端同步和团队空间。

| 第 2 周 | 实现学习计划创建、基础测评、路线图生成 | 表单、测评题、AI 生成接口、路线图展示 | 可运行 Demo | 用户能生成一份完整路线图 |

| 第 4 周 | 实现复盘调整、测试优化、MVP 验证 | 周复盘、AI 调整、异常处理、体验优化 | MVP 测试版 | 连续 7 天使用流程稳定，无关键阻塞 |

## 10.5 MVP 验证方式

| 验证问题 | 验证方式 | 判断标准 |
| --- | --- | --- |
| 是否能提升计划质量 | 对比手写计划和 AI 计划 | AI 计划更具体，包含阶段、任务、产出 |
| 是否能提升执行率 | 连续使用 14 天 | 每日任务完成率达到 60% 以上 |
| 是否减少资料选择成本 | 记录找资料时间 | 用户不需要额外大量搜索即可开始学习 |
| 是否能帮助复盘 | 每周完成复盘 | 用户能说清本周学了什么、卡在哪里 |
| 是否能判断掌握度 | 阶段测试和产出评估 | 用户能根据结果决定补学或进入下一阶段 |

# 12. 整体 PRD 验收标准

| 验收项 | 标准 |
| --- | --- |

| 计划调整 | 用户能延期、跳过、重排或应用 AI 调整建议。 |

# 13. 额外输出内容

## 13.1 产品一句话介绍

学径 AI 是一款面向个人学习者的 AI 学习计划 App，把模糊学习目标转化为清晰路线、每日任务、资料建议和复盘调整。

## 13.2 适合写进作品集的项目介绍

本项目是一款面向个人学习者的 AI 学习计划 App，聚焦用户在学习新领域时遇到的路径不清、资料过载、计划空泛、执行困难和效果难判断等问题。产品通过收集用户学习目标、当前基础、可用时间和学习偏好，利用 AI 生成阶段化学习路线图，并进一步拆解为每日任务、推荐资料、阶段产出和复盘机制。

目标用户包括希望快速入门 AI、数据分析、产品经理等方向的学生，准备转行或找实习的求职者，以及需要系统学习某一技能的职场新人。MVP 重点覆盖学习计划创建、基础测评、AI 路线图生成、每日任务、资料推荐、进度追踪和周复盘调整，暂不涉及商业化、社区、付费和多人协作。

项目的 AI 能力亮点在于：将用户自然语言目标结构化，基于基础测评生成个性化路线，根据执行情况动态调整计划，并通过笔记、测试和阶段产出辅助判断掌握度。该项目价值不在于替代学习本身，而在于提升个人学习计划质量、降低启动成本、增强执行反馈，并帮助用户形成持续复盘的学习闭环。

## 13.3 AI 产品经理求职项目包装建议

### 项目定位怎么写

建议定位为：面向个人学习场景的 AI 学习规划与执行管理工具，重点展示 AI 如何参与目标理解、计划生成、任务拆解、内容推荐、复盘调整和效果评估。

避免写成“万能学习神器”“颠覆教育行业”“全自动学会任何技能”等夸大表述。

### 简历项目经历怎么写

项目名称：AI 学习计划 App  
项目角色：AI 产品经理 / 产品设计负责人  
项目背景：个人学习者在进入新领域时常面临路径不清、资料过载、执行困难和学习效果难判断的问题。  
核心工作：  
- 梳理个人学习场景下的核心用户流程，设计从目标创建、基础测评、路线生成到每日任务和周复盘的完整闭环。  
- 设计 AI 路线图生成逻辑，将学习目标、基础水平、可用时间和偏好转化为阶段目标、任务、资料类型和产出物。  
- 制定 AI 生成内容风控规则，包括资源核验提示、用户可编辑、计划调整确认和不承诺绝对学习效果。  
- 规划 MVP 功能范围与 4 周开发节奏，明确 P0/P1/P2 优先级、页面结构和验收标准。  
项目成果：产出完整 PRD、核心流程图、页面结构和 MVP 验证方案，可支持后续原型设计和开发评审。  
能力体现：需求分析、AI 产品设计、Prompt/生成逻辑设计、移动端产品规划、MVP 范围控制、风险意识。

### 作品集重点展示内容

- 用户痛点和需求分层。
- 产品信息架构和核心流程。
- AI 功能设计表：输入、系统数据、处理逻辑、输出、风险控制。
- 路线图生成样例，如“30 天入门 AI 产品经理”。

- AI 生成内容的风险与边界。

### 面试时如何讲项目逻辑

可以按“问题 -> 用户 -> 方案 -> AI 能力 -> MVP -> 风险”的顺序讲：

1. 先说明个人学习最大的痛点不是没资料，而是路径和执行闭环缺失。
2. 再说明目标用户是有明确学习意图但缺少规划能力的个人学习者。
3. 然后讲产品如何把目标拆成路线、任务、资料、复盘。
4. 重点讲 AI 不只是聊天，而是参与结构化规划和动态调整。
5. 最后讲 MVP 为什么先做个人闭环，而不是社区、付费和增长。

### 如何突出 AI 产品能力

- 说明 AI 的输入不只是用户一句话，还包括基础测评、时间约束、偏好和历史执行数据。
- 说明 AI 输出不是一段建议，而是结构化结果：阶段、任务、资料、产出、验收。
- 说明你考虑了 AI 风险：幻觉、资源真实性、用户信任、计划过度承诺。
- 说明 AI 建议必须可解释、可编辑、可拒绝。

### 需要避免的表述

- 避免说“AI 自动让用户学会任何技能”。
- 避免说“资料推荐完全准确”。
- 避免没有数据依据地说“显著提升学习效率”。
- 避免把项目包装成已经大规模上线或商业化。
- 避免只讲模型能力，不讲用户流程和产品机制。
